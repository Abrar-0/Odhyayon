1.2 Scope of Project

This software will be a platform for teachers and students to engage in convenient learning. Teachers can create classroom, assign reading materials for students and assign deadlines for it. It will help the both parties to maintain a class in a disciplined manner.

Table: Teachers
Purpose: This table will hold the information of the teachers signing up for the service of Odhyayon. The stored information will be fetched during different interections a teacher makes with the interface of the website and the app.
Attributes:
ID: Used to give every teacher an unique identification number for the interface. ID will be used as the primary key for the table. This is autoincremented in the system and will assign an ID to each Teacher user on its own.
First_name: Used to store the teacher’s first name, which gets displayed on the interface when the teacher logs in.
Last_name: Used to store the teacher’s last-name, which gets displayed as a whole with the first_name when the teacher logs in.
Email: This attribute will store the email of the user and will be used as one of the main mediums of authentication for the user.
Password: This attribute will be used to store the password of the user, which will be stored encrypted. The password will be part of the main medium of authentication for the user, when logging in.
Date_of_birth: This attribute will store the date of birth of the user and will only allow users over the age of 13 to log in as a teacher in the interface.
Phone_number: This attribute will store the phone number of the user and will be used to help a user log in if they forget their usual login credentials.

Table: Students
Purpose: This table will hold the information of the students signing up for the service of Odhyayon. The stored information will be fetched during different interections a student makes with the interface of the website and the app.
Attributes:
ID: Used to give every student an unique identification number for the interface. ID will be used as the primary key for the table. This is autoincremented in the system and will assign an ID to each Student user on its own.
First_name: Used to store the student’s first name, which gets displayed on the interface when the students logs in.
Last_name: Used to store the student’s last-name, which gets displayed as a whole with the first_name when the student logs in.
Email: This attribute will store the email of the user and will be used as one of the main mediums of authentication for the user.
Password: This attribute will be used to store the password of the user, which will be stored encrypted. The password will be part of the main medium of authentication for the user, when logging in.
Date_of_birth: This attribute will store the date of birth of the Student and will allow teachers to know their students’ age.
Phone_number: This attribute will store the phone number of the user and will be used to help a user log in if they forget their usual login credentials.
Course_id: This is a foreign key that is collected from the table Courses. This attribute stores the ID of a course the student is enrolled in.
Profile_pic: This attribute is used to store a picture of the student, that will be used to authenticate the user’s access to resources of the class through face recognition.

Table: Courses
Purpose: This table will hold the information of the courses being created by a teacher. The stored information will be fetched during different interections a student and a teacher makes with the interface of the website and the app.
Attributes:
ID: Used to give every classroom an unique identification number for the interface. ID will be used as the primary key for the table. This is autoincremented in the system and will assign an ID to each classroom on its own.
Course_name: Used to store the name of the course, which gets displayed on the interface when the students or teachers are accessing this course.
Teacher_id:This is a foreign key that is collected from the table Teachers. This attribute stores the ID of the teacher who created by course.

Table: Uploads
Purpose: This table will hold the information of the materials being uploaded in a course by a teacher. The materials will be available for students when they check the dashboard of the classrooms.
Attributes:
File_id: This attribute creates an autogenerated unique ID for the files uploaded. This attribute is the primary key for the table.
Upload: This attribute stores the actual file that is uploaded. Files are fetched from here when students are accessing these files.
Uploaded_at:This attribute holds the time and date for the materials uploaded by a teacher.
Title:This attribute holds the title used when uploading the material.
Course_ID:This attribute stores the ID of the course this file is attached in. This attribute is a foreign key.

Table: Queries
Purpose: This table will hold the information of the queries attached by students, when they go through a certain material.
Attributes:
ID: The ID is the primary key of the queries, which give a unique identification number to each queries being made by a student.
Type: It is a composite key which holds the file format of the queries that are stored. When a question is asked in a text mode, the text is stored in a txt file, similarly when the queries are stored as voice messages, the queries are stored as mp3 file format.
Upload_id:This is a foreign key which stores the file ID of the file this query is attached to.
Student_ID:This is a foreign key that holds the ID of the student who has made the query.


6. Implementation Plan

    6.1 Approach

    The system is developed following waterfall model life cycle. The steps are requirement analysis,design, implementation and unit testing, integration and system testing and lastly maintenance. The system will be developed under super vision and rigorous testing to avoid bugs and meet the desired result.

    6.2 Design Pattern

    We are using MVC (MODEL-VIEW-CONTROLLER) design pattern provided by Django framework. MVC helps in avoiding mixture of code from different classes designed. It has a common database for android app and website that can be accessed through server remotely to create, view and update.

    • The model component- Handles the data and data flow of the system and performs various operations on data.
    • The view component- This component is responsible for how the data is represented to the user and how they interact with the data.
    • The controller component- Manages user interaction and passes these interaction to the view and the model.

    There are various ways in which to visualize and interact with system data. Same data can be represented differently based on different scenarios. Also when we will need to change the view of any specific part of the system, changing the view will suffice rather we would have to change the whole thing along with data. This will also make code more manageable as the various components are separated and in the same time inter-connected to ensure the date flow, connectivity between the components and full functionality of the system.
